## 태그 기능 구현 계획 (v1)

### 목표/범위
- **태그는 아티클별로만** 붙는다(카테고리처럼 “태그 사전” 테이블은 만들지 않음).
- 태그는 **간단한 요약 라벨** 용도이며, UI에서 제목 아래 **Chip** 형태로 노출한다.
- 태그 입력/수정은 **수동**으로만 제공한다(자동 태깅/Pro는 추후).
- **태그 최대 5개** 제한을 클라이언트/서버/DB 레벨에서 일관되게 강제한다.

---

### 데이터 모델(Backend / DB)
- **Article 테이블에 `tags` 컬럼 추가**
  - 타입: PostgreSQL `text[]` (SQLAlchemy `ARRAY(String)`)
  - 기본값: 빈 배열 `[]` (nullable=False)
  - 제약:
    - 서버에서 정규화 후 저장: trim, 공백 압축, 빈 값 제거, 중복 제거(대소문자 정책은 v1에서는 그대로, 추후 결정)
    - 길이 제한(예: tag 1~20자)과 개수 제한(<=5)
  - 인덱스(v1에서는 optional):
    - 태그 검색이 곧 필요하다면 `GIN` 인덱스 고려 (`tags`에)

---

### 백엔드 API 변경
- **아티클 생성(기존)**: 저장 링크 API에서 `tags?: string[]` 입력을 받도록 확장
  - 서버에서 정규화/검증 후 저장
- **태그 수정(신규)**:
  - `PATCH /articles/{article_id}/tags`
  - request body: `{ tags: string[] }`
  - response: 수정된 article(또는 tags만 반환). v1에서는 프론트 업데이트 편의상 article 반환 권장.
  - 권한: 현재 로그인 유저의 아티클만 수정 가능
- 스키마(Pydantic) 업데이트:
  - Article 응답에 `tags: list[str]` 포함

---

### 프론트엔드 UI 변경
#### 1) 아티클 추가 모달(`SaveLinkModal`)
- 입력란 추가: **태그(선택)**  
  - UX: 텍스트 입력 + Enter/Comma로 태그 추가, Chip으로 렌더링
  - 제한: 최대 5개, 중복 방지, 빈 값 방지, 길이 제한
  - 저장 시 API body에 `tags` 포함

#### 2) 아티클 더보기 메뉴(…)
- 메뉴에 **“태그 수정”** 항목 추가
- 클릭 시 **태그 수정 모달** 오픈

#### 3) 태그 수정 모달
- 기존 태그 로드 → Chip 편집 UI 제공(추가/삭제)
- 저장 시 `PATCH /articles/{id}/tags` 호출
- 성공 시 목록/카드 즉시 업데이트(로컬 상태 갱신 또는 refetch)

#### 4) 리스트/카드에서 태그 노출(3번째 이미지)
- 제목 아래에 `Chip` 리스트로 태그 렌더링
- 태그가 없으면 영역을 렌더링하지 않음

---

### 구현 순서
1. Alembic 마이그레이션: `articles.tags` 컬럼 추가
2. Backend:
   - 모델 `Article`에 `tags` 추가
   - 스키마/서비스/라우터에서 create 응답/입력에 tags 반영
   - `PATCH /articles/{id}/tags` 구현 + 검증/정규화
3. Frontend:
   - 타입(`ArticleCard` 등)에 `tags` 추가
   - `SaveLinkModal`에 태그 입력 UI + create 요청에 tags 포함
   - 더보기 메뉴에 “태그 수정” + 모달 생성 + API 연동
   - 아티클 리스트/카드에 Chip 렌더링
4. 제한/에러 처리:
   - 태그 5개 초과 시 즉시 안내(프론트) + 서버도 400으로 방어
5. 확인:
   - 새 아티클 저장 시 태그 저장/노출
   - 태그 수정 후 즉시 반영

---

### 결정/가정(명시)
- v1에서는 태그의 대소문자 정규화(예: lowercasing)는 하지 않고, trim/중복만 처리한다.
- 태그 검색은 “입력 필드에서 검색 가능” 요구가 생기면 GIN 인덱스와 쿼리 확장을 추가한다(이번 작업 범위에서는 UI/저장/수정/표시까지만).
